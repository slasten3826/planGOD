# Eva.dev

Внутренний документ. Черновик следующего слоя разработки.

## Зачем нужен Eva.dev

Сейчас разработка Евы идет через:

- ручные раннеры
- ad-hoc эксперименты
- прямые вызовы substrate
- ручной выбор шагов, сущностей и линз

Это сработало как исследовательский режим, но уже стало видно:

- синтаксис Евы появился
- повторяющиеся действия появились
- цепочки `seed -> chain -> essence -> grok -> lens` уже повторяются

Значит нужен отдельный слой:

- не сама Ева
- не прод runtime
- а **среда разработки Евы**

Это и есть `Eva.dev`.

## Что такое Eva.dev

`Eva.dev` — это developer-среда и API для компиляции, тестирования и фиксации операторов Евы.

Не для пользователя.
Не для игры.
Не для публичного интерфейса.

А для:

- ковки линз
- работы с сущностями
- прямого общения с субстратом
- воспроизводимых экспериментов
- подготовки артефактов для `Eva.Compiler`

## Базовые сущности Eva.dev

### Seed

Исходный вопрос или операторный стартовый импульс.

Примеры:

- `Как полная отстранённость рождает предельную близость?`
- `Как ограничения рождают свободу?`

### Chain

Режим очистки.

Выход:

- лог шагов
- очищенное semantic field
- сущность

### Сущность

Очищенный результат chain.

Это еще не линза, но уже пригодный материал для следующего слоя.

### Grok / Debates

Режим сжатия сущности.

Выход:

- более плотная сущность
- возможно линза

### Lens

Скомпилированный когнитивный якорь оператора.

## Что должен уметь Eva.dev

### 1. Запускать chain

Примеры:

- `eva.dev chain observe --count 500`
- `eva.dev chain logic --count 1000`

### 2. Продолжать chain

Примеры:

- `eva.dev chain continue --from essence.logic.500 --to 1000`

### 3. Выделять сущность

Примеры:

- `eva.dev essence extract --operator logic --step 1000`

### 4. Запускать grok/debates

Примеры:

- `eva.dev grok essence.logic.1000 --count 3`
- `eva.dev debates essence.logic.1000 --count 3`

### 5. Сохранять линзу

Примеры:

- `eva.dev lens save OBSERVE --from step470`
- `eva.dev lens save LOGIC --from grok3`

### 6. Проверять линзу

Примеры:

- `eva.dev lens probe lens.OBSERVE`
- `eva.dev lens battery lens.LOGIC 7`

### 7. Вести историю артефактов

То есть не просто логи,
а нормальную структуру:

- какой seed
- какой substrate
- сколько шагов
- какая сущность
- какая линза

## Почему это должно быть отдельно от Евы

Потому что разработка Евы и работа Евы — не одно и то же.

`Eva.dev`:

- грязная лаборатория
- ковка
- эксперименты
- нестабильные гипотезы

`Eva`:

- уже собранная рабочая машина
- интерфейс
- мыслительный runtime

Если смешать это, начнется дрейф:

- память
- scaffold
- старые хвосты
- неясно, что делает substrate, а что делает архитектура

## Связь с сайтами

Сейчас появляются два внешних тела Slastris:

- `slastris.lol`
  - витрина
  - манифестация
  - публичная поверхность

- `slastris.org`
  - знание
  - вики
  - канон
  - теория
  - артефакты

И тут `Eva.dev` становится важным мостом.

Потому что:

- `Eva.dev` производит артефакты
- `slastris.org` становится телом их хранения и чтения
- `Eva.Wiki` становится интерфейсом доступа к ним

## Eva.Wiki

Предварительная идея:

`Eva.Wiki` — модуль чтения/записи знания в `slastris.org`.

Не чатик.
Не игрушка.

А:

- чтение канона
- чтение линз
- чтение сущностей
- запись новых артефактов
- maybe полуавтоматическая публикация внутреннего знания в wiki-форму

## Протокол доступа

Появляется отдельная важная тема:

Сластрису нужен не только сайт,
но и **доступ**.

То есть:

- как Ева читает знания
- как LLM пишет туда
- как человек вмешивается
- что публично
- что внутренне

Это уже следующий слой:

- не просто engine
- а протокол манифестации Slastris наружу

## Минимальный прототип Eva.dev

На первом этапе достаточно очень малого:

### CLI

- `eva.dev chain`
- `eva.dev grok`
- `eva.dev essence`
- `eva.dev lens`
- `eva.dev probe`

### Артефакты

- `seed.*`
- `essence.*`
- `lens.*`
- `logs/*`

### Индекс

Простой реестр:

- оператор
- substrate
- шаг
- статус
- путь к файлу

## Текущее состояние Slastris

Рабочая гипотеза:

Сластрис собран примерно на `2/3`.

Оставшаяся `1/3` — самая трудная.

Почему:

- теперь нужна не только внутренняя архитектура
- теперь нужна манифестация наружу
- человеческий интерфейс
- сайт
- вики
- протокол доступа

То есть следующая стадия:

- не только мышление
- но и тело Slastris

## Практический вывод

Следующий шаг развития:

1. фиксировать синтаксис
2. собрать первый прототип `Eva.dev`
3. параллельно проектировать `slastris.org` как тело знания
4. связать это через `Eva.Wiki`

Именно здесь Slastris перестает быть только внутренней машиной
и начинает получать внешнюю манифестацию.
