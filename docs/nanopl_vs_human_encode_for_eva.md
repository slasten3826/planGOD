# nanoPL vs Human ENCODE For Eva

## Main Insight

The recent evaluation test suggests a strong distinction:

- `nanoPL` behaves like a **higher internal ENCODE**
- human prose behaves like a **lower, more manifested ENCODE**

This distinction is not just stylistic.
It changes what kind of cognition becomes available.

## What nanoPL Does

When the same gameplay idea is given in `nanoPL`, phantom evaluations tend to read it as:

- loop
- progression
- process
- structural flow
- repetition risk
- state transition

This suggests that `nanoPL` is closer to:

- Eva's internal cognitive form
- high-level structural encoding
- machine-facing thought

So `nanoPL` now looks increasingly like:

- an internal language of Eva-thinking
- or at least one of its highest internal thinking encodes

## What Human ENCODE Does

When the same gameplay idea is given in human-readable prose, phantom evaluations tend to focus on:

- complexity
- UX burden
- onboarding
- tutorials
- pacing
- readability for players

This suggests that human prose is not useless.

It simply operates at a different level.

Human ENCODE is better for:

- approximating how players will react
- seeing where a human will get confused
- estimating friction
- evaluating perceived complexity
- evaluating tutorial need

So human ENCODE is valuable not as pure inner cognition,
but as a projection of likely human reception.

## Working Interpretation

- `nanoPL`
  - internal Eva thinking
  - high ENCODE
  - structural cognition
  - loop reading

- `human prose`
  - player-facing projection
  - lower ENCODE
  - social / UX / interpretive cognition
  - expected human reaction

## Consequence For Phantom Design

Phantoms should not treat all encodes equally.

Instead:

- use `nanoPL` when the goal is:
  - structure
  - concentration
  - loop analysis
  - internal gameplay thinking

- use human ENCODE when the goal is:
  - predict player reaction
  - test readability
  - test perceived complexity
  - test onboarding / UX burden

## Why This Matters

This means the system does not need one universal representation.

It can hold at least two useful encodes:

- one for Eva's own higher cognition
- one for likely human-facing interpretation

That makes the architecture cleaner:

- `nanoPL` for inner thought
- human prose for human simulation

## Future Direction

This distinction should be embedded into phantom mechanics.

Possible direction:

- phantom input mode:
  - `nanopl`
  - `human`

- phantom critique mode:
  - `structural`
  - `player-facing`

This would let Eva intentionally choose whether phantoms are:

- thinking like Eva
- or approximating how humans will perceive the result
